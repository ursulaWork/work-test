Classification Definition
Adopted Notations
User Story Element <definition>
The story element should have ‘Story’ in the title or metadata to identify it as a story.
The story element utilises the Use Case element in the form of the rectangularelement notation.
Acceptance Criteria
Acceptance Criteria are identified by stakeholder.
Each acceptance criteria is documented individually.
Its textual description is held in the metadata.
It normally follows the syntax:
Given…
When…
Then...
Priority
The importance the stakeholders place on a requirement. MoSCoW Prioritisation is used.
Priority should be set in the metadata.
The value should be the same as the user story notes.
Story Source and Input
It is recommended to note the stakeholders involved (usually the product owner). Tagged values are used to capture this requirement metadata.
Tagged values types and metadata for:
Originator
Story Classification
The status indicates the current status of a the story for the project. The default values are: proposed, approved, implemented and deprecated.
The metadata status is set to the agreed values for the project.
Story Alias
This is an alternative identifier for a Requirement element.
Story Stereotype
It is recommended to stereotype the Use Case element to identify it as a story to distinguish them from use case.
