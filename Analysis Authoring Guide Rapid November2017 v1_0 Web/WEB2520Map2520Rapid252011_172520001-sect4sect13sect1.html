This section outlines the adopted notations for creating an Information Analysis Model (IAM) Artefact (for an example of an IAM, see the figure: information analysis model — example diagram).
The following rules and guidelines apply to the adopted notation for IAM Diagrams:
•  The IAM is located in a package named ‘Information Analysis Model’ within the ‘Shared artefacts’ package (in the Areas of Interest package, in a package labelled 'Information Analaysis Model’ within the Area of Interest top-level package).
•  IAM’s shall only be developed in accordance with the Adopted Notations of the Approved Methodologies.
–  They must also adhere to the standard Core Notations for UML Class Models.
•  These IAM specific adopted notations should also be read in conjunction with the Generic Class Model Artefacts Adopted Notations.
•  The Information Analysis Model uses the Business Information Model (BIM) as its input.
–  A class in the BIM that falls within the Area of Interest must be elaborated (by one or more classes) on the IAM, and every IAM class must be derived from a single class in the BIM.
•  IAMs must only be developed using the currently approved CASE Toolset or documentation approach.
•  It is rare for multiple IAMs to be developed within a single project; however more than one may exist within a domain that focuses on separate Areas of Interest.
–  Where there is overlap between the areas of interest, separate models should be consolidated.
–  Large or unwieldy IAM Diagrams may also be split, using ‘Composite’ Class Diagrams or Extract Diagrams to aid navigability / readability, in accordance the chosen Methodology’s defined approach.
•  The IAM Artefact represents Class Elements and their Associations (with directions).
–  Class Elements:
♦  Must have a name (unique, clear and unambiguous; using camel case formatting convention e.g. ‘investigationType’).
♦  A description of the Class and its purpose in the metadata notes.
♦  No Operations are to be shown for classes within the IAM.
♦  Class element may also have Attributes applied to them (these may also show on diagrams), attributes:
◊  Must have a names (unique, clear and unambiguous; using camel case formatting convention e.g. ‘investigationType’).
◊  All attributes must have specified Datatypes, which must conform to the standard list of Datatypes (where determined).
◊  Multiplicities may also be applied Attributes (but not typically where the cardinality is ‘1..1’).
◊  Constraints.
◊  A brief description of their purpose.
–  Associations on Class Diagrams:
♦  Association Multiplicities may also be applied upon Class Diagrams, at each end of the connector (and typically where cardinality is not ‘1..1’).
♦  Directions (where applicable).
♦  Names (where appropriate).
♦  Must not have Constraints.
♦  Where associations cross on a Diagram, these lines must be at right-angles to each other.
♦  Multiple associations between two classes are prohibited.
•  The use of a foreign key (i.e. identifier used to extract information from external systems such as PDS and SDS) in place of the full information requirement of a class is prohibited:
–  For example, where full demographic information is the requirement, this shall not be substituted with NHS number; this is to validate any future-referenced system functionality fulfils the information requirements.
•  It is not expected that any systems will appear as Classes on the IAM.
•  Data concepts within the Use Case Model are candidates for representation within the IAM as a class or attribute.
•  All of the information required to fulfil the business processes captured in the Use Case Model Artefact should be defined in the IAM, therefore these artefacts should be produced in parallel.
