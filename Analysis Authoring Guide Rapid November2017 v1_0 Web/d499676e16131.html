User Stories are identified during Discovery Stage of an analysis project. The initial story detail is kept brief and is added to as the project progresses based on its importance to the product owner.

The business analyst checks the detail of the user story: 
• 
Stories is in the following format as agreed with the Developer and Tester:
◦ 
Syntax:  As a [role],
◦ 
I MUST/SHOULD/MAY to  [action],
◦ 
so that [goal].
• 
Stories contain the relevant constraints, usually as Acceptance Criteria.
• 
Stories contain information on priority : MUST, SHOULD, MAY.
• 
Additional information can be added as notes.
Invest Criteria
The INVEST criteria can be used as guidance to define a good user story:
• 
I – Independent.
• 
N – Negotiable.
• 
V – Valuable.
• 
E – Estimable.
• 
S – Small.
• 
T – Testable.
Large and / or complex stories may need to be broken down before they can be included in a sprint backlog. The developers and testers can advise as to whether a story is a suitable size.
The Business Analyst can also consider elaborating the user story as a UML Activity Diagram and/or sequence diagram to aid team understanding. Data items can be described using a UML Class Diagram.
