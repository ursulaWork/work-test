Data Products currently utilise two distinct Analysis Methodologies. Analysts typically choose which to adopt early in an analysis project (for further details see the subsection: Choosing an Analysis Methodology and Approach).
Methodology Description
Generic Methodology
This methodology follows a formal structured approach to model development .
Rapid Methodology
This methodology follows an ‘Agile’ approach to model development .

Choosing an Analysis Methodology and approach
Choosing an Analysis Methodology and approach
A key aspect when choosing which methodology to use is to consider the Programme / Project Delivery Approach. The governance of a project or programme defines how the initiative is set up, controlled and managed. The Lead Business Analyst and / or Project Manager should assess a project’s needs before deciding which methodology to adopt (see the subsection below: Assessing the project fit with the Agile approach).
Programme / Project Delivery Approach
The delivery approach of the programme or project needs to consider the governance framework and its fit with each of the Data Products Analysis Methodologies. This is influenced by the management of the domain and the arrangements for reporting its status reporting to the programme or project board.
The key factors that will be key to determining the delivery approach are ‘Time’, ‘Quality’, ‘Cost’ and ‘Features’. The factors that are fixed will indicate the most appropriate approach.
For example: if ‘Features’ and ‘Quality’ are not negotiable (but ‘Time’ and ‘Cost’ are variable), then a more traditional delivery approach is  appropriate. The Generic Analysis Methodology would be the recommended approach in this instance.
Where ‘Time’, ‘Quality’ and ‘Cost’ are fixed (but ‘Features’ are variable), then a more Agile like approach would be appropriate. The Rapid Analysis Methodology would be the recommended approach in this instance.

Hint: The Business Analyst has a key role to assist the project with:
•  Assessing Project Fit with Agile Approach.
•  Production of the Analysis Scope and Objectives Document.
•  Ensuring prioritisation of requirements and user stories.
•  Ensuring the acceptance criteria for “done / completeness” (in Agile projects the Product Owner decides when a project is complete).
Assessing the Project-Fit with an Agile approach
A realistic evaluation of the project working environment should help the project executive understand the commitment needed and risks if choosing an agile approach for the project or programme.
Who
Project Manager, Business Analyst, Project Executive and key stakeholders.
Why (purpose)
The purpose is to assess if the programme or project can be run effectively following Agile principles and that the Rapid Authoring Guide should be followed by the Business Analyst.
When
At the start of a project or programme - inception stage.
What
The Project Manager and Business Analyst will need the Project Executive and key stakeholders to answer some questions regarding the domain.
The findings, recommendations and risks are reported back to the project executive so they can make an informed decision on whether the project could follow an agile approach.
The business analysis could use a project approach questionnaire and document the findings in the project risk register and Analysis Scope and Objective Document.
Outputs
Finding and recommendations for project executive.
Analysis Scope and Objectives Definition document.
Risks.
