This section outlines the Generic Core Notations for Constraints captured against an element (diagrammatic representation of a constraint on a diagram is dependent upon the type of constraint being used, example figures are available in the definitions for the various types of Constraint, as applicable).
- Where model authors are developing artefacts that utilise constraints, they should also be aware of the generic Constraints.
- At some point in the future the use of more structured ’machine-readable‘ constraints such as Structured English, OCL or RegEx may be utilised in addition to narrative constraints.
The following outlines generic Core Notations for Constraints:
•  Constraints should only be captured within the area of the metadata / documentation for an element prescribed for Constraint capture (this is usually achieved using a CASE Toolset).
–  For consistency, ease of use, locating and displaying of Constraints, it is important to capture them in the same place across all Models. This is especially important where a model is being revised from a previous version or has a new author.
•  This content should also be read in conjunction with any descriptions provided as part of each specific type of Constraint and Artefact.
•  Constraints should not seek to capture or replicate information that is more appropriately captured elsewhere within the Model by other functions or annotations, e.g. where they are Requirements.
•  It is advised that as few formal Constraints as possible are displayed on diagrams, to avoid diagram clutter and confusion with other annotation types.
•  Where constraints pertain to another Element of the Model, that element must be referenced:
–  Referencing another Attribute should be done using the full and exact Attribute name.
–  Referencing another Element should be done using the alias of that Element (in some CASE toolsets this may be an auto-generated number or identifier).
•  All Constraints recorded must have:
–  A short, meaningful title.
–  Narrative defining the constraint (written in everyday readable English).
–  Constraint type.
•  Datatypes are also a type of Constraint (further details can be found in the Core Notations for Datatypes).
