The following outline the Core Notations for Packages within models:
Entity
Definition / Context
Core Notations
Structural Model Packages
“A Package within a model is a folder that holds a collection of related entities.”
The Package must have a meaningful name - acronyms are NOT to be used when naming packages.
A Package may contain sub-folders.
The default packages (names and structure) are prescribed in the Analysis Model Template.
Catalogue Package
“A Catalogue Package is a folder containing the same type of elements.”
The Catalogue Package must have a meaningful name which follows the format: ‘<<element type>> Catalogue’ (e.g. Actors Catalogue)
The Catalogue Package should be held in a suitable model location (use of Analysis Model Template indicates currently defined Catalogue Packages).
A Catalogue Package may contain sub-packages (with meaningful names).
A Catalogue Package may be a sub-package of another model package (e.g. within a Use Case Artefact / Model Package).
Package Diagram (and package elements on diagrams)
“A Package Diagram is a visual representation of packages in a model”
It can be used to represent:
•  Model hierarchy.
•  A logical representation of related packages (for example to illustrate a "theme").
It can include associations between packages.
Packages may also be represented on other diagram-types as Package Elements.
The Package Diagram should have a suitable name.
The Association types between packages on a Package Diagram can include:
•  Dependency.
•  Realise.
•  Trace.
A Package Element represented upon a diagram may also show any contained elements or sub-packages within it (typically only at the first-level — CASE Toolset dependent).
