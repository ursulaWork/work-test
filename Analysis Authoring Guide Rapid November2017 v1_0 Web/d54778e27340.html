Actor Element
tablespacer
tablespacer
After p in entry
Actor Elements define the various roles (human or system) that interact with Use Cases; to trigger and / or to receive input(s), from individual Use Cases.
After p in entry
Actors are represented on UML Use Case Diagrams using a stick-person symbol.
The name of the actor should be meaningful in the context of the domain.
Actor hierarchy can be implied using the Generalise / Generalization relationship.
Where an Association goes from an Actor to a Use Case element this indicates that the Actor plays a role in performing the Use Case.
Actors are related to other entities on diagrams.

Boundary Regions
tablespacer
tablespacer
After p in entry
A Boundary Region on a Use Case Diagram encloses all the Elements that represent the ‘system’ functionality being modelled. It also defines the point were roles (Actors) interface with the system
After p in entry
A Boundary Region is represented as a rectangle shape use to represent the system; it can be labelled to describe the context for the boundary
Use Cases are held inside the boundary region. whereas Actors are placed outside of it.
The use of Boundary Regions is optional.

Associations on Use Case Diagrams
(example figures are shown against each usage)
After p in entry
UML Use Case Diagrams utilise a range of Associations (as connectors) to define the nature of the relationship between Use Cases, as well as with or between Actor Elements.
After p in entry
An Association is shown as a line between two diagram elements (e.g. Use Case and Use Case, Actor and Actor, or Actor and Use Case).
An Association may also be labelled as well as indicating the direction in which it should be read. Notations will vary according to Association type.
Constraints may NOT be recorded against Associations.
After p in entry
Undefined or ‘Standard’ Association
tablespacer
tablespacer
After p in entry
An Undefined or ‘Standard’ Association indicates a a relationship between the connected Elements (e.g. Use Case and Use Case, Actor and Actor, or Actor and Use Case).
Where the Association is between an Actor and a Use Case, it indicates that the Actor plays a role in performing the Use Case
After p in entry
An Undefined or ‘Standard’ Association is represented as a plain line between the two connected Elements (it has no further notation).
After p in entry
Include Association
tablespacer
tablespacer
After p in entry
An Include Association between Use Case Elements indicates that one Use Case contains the functionality of another use case as part of their normal processing.
After p in entry
The Include Association is represented by a dotted line with an open arrowhead (pointing to the Use Case to be included).
The line is labelled <<include>> .
After p in entry
Extend Association
tablespacer
tablespacer
After p in entry
Extend Associations indicates that one Use Case can extend the behaviour of another, typically when exceptional circumstances are encountered.
After p in entry
Extend Associations are represented by dotted line with an open arrowhead (pointing from the Use Case that contains the extra behaviour to the main Use Case).
The line is labelled <<extend>>.
The conditions under which this piece of processing is to be triggered should be stated, preferably as a Business Rule.
After p in entry
Generalise / Generalization Associations
tablespacer
tablespacer
After p in entry
A Generalise / Generalization Association is used to indicate inheritance. Drawn from the specific classifier to a general classifier, the generalization's implication is that the source inherits the target's characteristics.
A Generalise Association on a Use Case Diagram can be used to generalise Actors or Use Cases:
• 
To share or ‘inherit’ the association relationships of parent element.
• 
A ‘child’ element to specialize a parent element.
Depending upon in which direction it is read it the relationship becomes either a ‘Generalization’ or a ‘Specialization’ Association.
After p in entry
Generalise / Generalization Associations are represented using a solid line with a filled arrow, pointing to the generalised element.
Generalise / Generalization Associations can only be used to point to the same type of element (e.g. between Actors or between Use Cases — but NOT between Actors and Use Cases).
See below for separate notations defining Generalization Associations between Actors and Use Case Elements
After p in entry
Generalise / Generalization Associations - between Use Cases
tablespacer
tablespacer
After p in entry
A Generalise / Generalization Association between Use Cases identifies where the behaviour of the specialised Use Case replaces the behaviour of the generalised Use Case.
The Generalised Use Case could also be an considered an abstract Use Case that holds all the common behaviours.
Generalization is used when you find two or more use cases that have commonalities in behaviour, structure, and purpose. The Specialised Use Cases holds the more specific additional behaviours — the general goals for each Use Case should be the same
After p in entry
Generalise / Generalization Associations are represented using a solid line with a filled arrow, pointing to the generalised Use Case Element.
After p in entry
Generalise / Generalization Associations — between Actors
tablespacer
tablespacer
After p in entry
A Generalise / Generalization Association between actors illustrates the differences in responsibilities of the users of a system. The ‘specialised’ Actor can perform all the activities of the ‘generalised’ role as well as any specific aspects of its own role.
After p in entry
Generalise / Generalization Associations are represented using a solid line with a filled arrow, pointing to the generalised Actor Element.
After p in entry
Realization / Realize Association
tablespacer
tablespacer
After p in entry
A Realization / Realize Association identifies where a Use Case Element ‘realises’ or implements the goals / aims or specifications of a non-Use Case (e.g. where a Use Case realises a Requirements Element).
Realization / Realize Associations can be created between a Use Case and other types of elements.
After p in entry
A Realization / Realize Association is represented as a dashed line with a solid-filled arrowhead.
The source element realises the destination element (e.g. the one it points to).
The line is labelled <<realises>>.

Use Case Element (and Use Case Constraints)
tablespacer
tablespacer
After p in entry
Use Case Elements individually define a discrete process. The completion of that process should return an observable result.
After p in entry
Use Case Elements are represented as an ellipse with the name of the Use Case inside it.
The name should be a meaningful expression (verb / phrase) in the context of the domain.
A Use Case Element’s metadata may also contain:
NOTE: - The content relating to Use Case metadata (e.g. Constraints) may or may not be shown on Diagrams, as required.
• 
Textual description.
• 
Constraint information:
◦ 
Triggers, Goals, Pre and Post Conditions.
• 
Definitions for Main and Alternate Paths.
Core Notations are also available for UML Use Case Diagrams (see the subsection: Use Case Diagram [Core Notations])

Composite Use Case Element
tablespacer
tablespacer
After p in entry
Composite Use Case Elements denote subordinate behaviours; these are elaborated on a separate diagram.
After p in entry
Composite Use Case Elements are represented as Use Case Elements that include a "spectacle" symbol.
This Element links to a subsidiary diagram which further elaborates the Use Case.

Use Case - Basic Path
(no figure - only visible as metadata)
After p in entry
The ‘Basic Path’ for a Use Case defines the usual steps that take place in order to achieve its intended goal.
After p in entry
Basic Paths are textually defined metadata statements captured within the element.
Label as: "basic path".
Basic Paths can also be additionally described by a Scenario or as part of an Activity Diagram.
After p in entry
Use Case - Alternate Path
(no figure - only visible as metadata)
After p in entry
The ‘Alternate Paths’ for a Use Case defines when a use case diverts from the basic path. Alternate Paths define the processing to be undertaken when a known or planned event causes a deviation from the Basic Path.
After p in entry
Alternate Paths are textually defined metadata statements captured within the element.
Label as: "alternative path".
The Alternate Paths should also include the business rule that indicates the condition that causes the deviation from the basic path.
Alternate Paths can also be additionally described by a Scenario or as part of an Activity Diagram.
After p in entry
Use Case Constraint - Trigger
(no figure - only visible as metadata)
After p in entry
A Trigger Event documents the stimulus that is required to cause a Use Case to commence its processing / functionality.
Use Cases will normally have only one Trigger event; however if more than one Trigger is identified then care is required and several questions must be addressed, such as:
• 
Do all the Trigger events need to have occurred before the use case can start?
• 
Does each Trigger lead to exactly the same flow(s)?
• 
Is there an order in which the Trigger events need to occur?
Triggers may be identified as the result of the invocation of a business rule.
After p in entry
Triggers are textually defined metadata statements captured within the element.
Label as a Constraint - type: ‘trigger’
After p in entry
Use Case Constraint- Goal
(no figure - only visible as metadata)
After p in entry
A ’Goal’ for a Use Case or User Story states its defined aim upon completion. When the processing completes successfully, the stated goal must have been achieved
After p in entry
Goals are textually defined metadata statements captured within the element.
Label as a Constraint - type: ‘goal’
After p in entry
Use Case Constraint - Pre-Condition
(no figure - only visible as metadata)
After p in entry
A Pre-Conditions is something that must have happened or a specific state of something prior to that Use Case or User Story commencing its processing. The Use Case or User Story cannot start until all stated Pre-Conditions are true.
After p in entry
Pre-Conditions are textually defined metadata statements captured within the element.
Label as a Constraint - type: ‘pre-condition’
After p in entry
Use Case Constraint - Post-Condition
(no figure - only visible as metadata)
After p in entry
Post-Conditions define the outcomes that are to exist when processing is concluded.
There are two types of Post Condition:
• 
A ‘Successful’ Post-Condition is the outcome when processing runs as planned.
• 
A ‘Guaranteed’ Post-Condition is the outcome when processing encounters an error.
After p in entry
Post-Conditions are textually defined metadata statements captured within the element.
Label as a Constraint - type: ‘post-condition’
