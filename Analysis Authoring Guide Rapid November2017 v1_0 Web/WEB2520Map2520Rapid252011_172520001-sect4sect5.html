“A range of Requirements Artefacts are defined to support the documentation and management of requirements within an Analysis Model; these are defined for each Methodology. Requirements are documented in the analysis model using individual Requirements Elements which are organised into sub-Packages in the Requirements Catalogue. Requirements Elements and the relationships between them are elaborated on Requirements Diagrams.”
- For further details of the various artefacts and packages that can be developed to support requirements, see the subsection: Requirements Artefacts and Packages.

Requirements artefacts are key to:
•  Defining the scope for a project and/or product.
•  Supporting the development of other content in the analysis model.
Key stakeholders should be fully involved throughout the lifecycle of a project in the development and validation of requirements.
In addition, a number of activities and considerations also support the creation and development of requirements and the elements that elaborate them (for further details see the section: Developing Requirements), including:
•  Eliciting requirements,
•  Requirements characteristics,
•  Classifying and organising requirements.
A project following the Rapid Methodology should strive to keep the requirements ‘high level’. This should ensure that the project is focussed and can embrace changes through the use of detailed type requirements such as user stories etc.,.
The modelling approach for Requirements outlined in this section is based heavily on the functionality of the Enterprise Architect CASE Toolset; however it can also be applied to any toolset / documentation approach.

Requirements Artefacts and Packages
Requirements Artefacts and Packages
The following outline the context and relationships between the various requirements packages:
•  About this Document — This holds a document that is a report generated from the Requirements Catalogue within the Analysis Model.
•  How to use this Document — This package should include a document (created within the Model) outlining the background information regarding how to use and interpret the requirements artefacts within the Analysis Model.
•  Background — This should hold documentation providing background information on the project / business area / domain that is the subject of this Analysis Model.
•  Requirements Catalogue  — This package contains all of the organising sub-packages, diagrams and elements for the formal Requirements Artefacts within the Analysis Model. It is held within the ‘Requirements Package’ at the top-level of the model:
- Projects that need to elaborate requirements for more than one business subset / domain within the same model may add individually labelled Requirements Catalogues to the Requirements Package (as well as optionally as a ‘Shared Requirements Catalogue Package’, where requirements apply to more than one subset) — these should have the same content classifying sub-packages and artefacts created within them
–  Requirements Catalogue Diagram — This is a ‘Package Diagram’ that represents the organising packages (e.g. ‘Functional’ / ‘Nonfunctional’) held in the Requirements Catalogue Package (within which it is also located) as Package Elements.
–  Requirement Catalogue Sub-packages:  — Used to organise and classify requirements within the Analysis Model; each distinct sub-package includes the following content: 
♦  Requirements Diagram(s) — Are a formal artefact, used to elaborate a single set of individual Requirements Elements as well as any relationships between them, using Associations (Requirements Diagrams may also display related Non-Requirements Elements — e.g. Use Cases).
♦  Requirements Elements — Are used to instantiate individual requirements within the model. They are primarily classified by their ‘owning’ parent package (as either Functional or Non-functional).
Non-diagrammatic Requirements Outputs — outputs (in RTF or CSV formats) detailing the content of Requirements Elements may be generated from the Analysis Model; these outputs are classed as Requirements Artefacts:
- A copy of any output generated from the Analysis Model’s content must always be re-imported back into it; these are held as Related Documents within the model
