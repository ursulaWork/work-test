“Requirements Elements are used to articulate single requirements within an Analysis Model.”
The following table describes the information that is held for each requirement element when produced as part of a formal requirement product.

Adopted Notations 
Requirements Element: Name
A short description of the requirement.
The textual descriptions are included in the metadata as follows:
•  The short name to be prefixed by a unique identifier; the format is <project shortcode>_ <unique number>_ <REQ>  followed by a short description.
•  The identifier should be the same as the alias.
•  The long textural description to include the priority in the text.
•  Where an element is deprecated in which case the word ‘Deprecated’ MUST appear before the unique identifier.
Requirements: Notes
These notes fully describe the requirement.
The textual note should include:
•  The priority in the wording uses  one of  ‘MUST’, ‘SHOULD’ or  ‘MAY’.
•  Can  further classify any operational, legal, performance or reliability aspects etc.
•  It can describe the decision for a requirement that has been deprecated.
•  It can include references to any external documents, URLs  etc.
Requirements: Alias
This is an alternative identifier for a Requirements Element.
Metadata to include a unique identifier. 
Requirement: Priority
The importance the stakeholders place on a requirement.  MoSCoW prioritisation is used.
Priority should be set in the metadata.
The value should be the same as the requirement notes. 
Requirements: Classifications
The following items outline some of the classifications that can be applied to Requirements.
Requirements: Classification
Metadata are used to capture any classification for the type of requirement such as ‘Functional’ / ‘Non-functional’.
Requirement metadata indicates if a requirement is Functional or Non-Functional requirement.
 
Requirements: Classification
Requirements status indicates the current status of a requirement for the project. The default values are: proposed, approved and deprecated.
Requirement metadata status is set to the agreed values for the project.
Requirements: Classification
Themes or sub-domains can be used to ‘logically’ organise Requirement Elements in a complex domains.  
Package notation with suitable namespace name that reflects the theme.
The package should include  a brief description of the theme in the package metadata.
Requirements: Classification
Stakeholders involved in the requirement gathering:
•  Originator —The person / organisation originating or raising the requirement, as well as the date it originates from.
•  Sponsor — The person/organisation sponsoring or supporting the need for this requirement. 
•  Contributors — The person / organisation contributing to the definition and resolution of this requirement and the date/s of their contribution.
Tagged values types and metadata for:
Originator, Sponsor, and contributor (optional).
The person or organisation.
The date of contribution.
 
Requirements: Classification
Project stage: the project may have a number of stages for implementation of the solution. The Requirement Element can be assigned to a specific stage in a project (if known).
Requirement metadata for stage is set (the default is 1.0).

ArtReqElmntDeprecateAN Deprecating Requirements Elements (Adopted Notations)
This section outlines the Adopted Notations for Deprecating Requirements Elements (an example figure is not available for this content):
•  Requirements Elements that are no longer realised by a model or are redundant MUST NOT be deleted from the Analysis Model, but must instead be ‘Deprecated’. This is to enable a historical understanding of the context and lifecycle for a specific requirement to be referenced whenever the model is reviewed, or to support further development if the model is reactivated at a later date.
•  The deprecated Requirements Element must be moved from its classifying package (e.g. ‘Functional’ / ‘Non-Functional’) within the Requirements Catalogue Package into the top-level ‘Deprecated’ Package.
•  Deprecated Requirements MUST also adhere to all of the Rules and Guidelines for Deprecating a Model Entity, as defined by the Approved Methodologies:
–  Additionally within the Requirements Model:
♦  Deprecated Requirements Elements must be removed from any active Requirements Diagrams:
◊  When moving Requirements Elements for deprecation from a diagram, Model Authors must ensure that any valid links (including Associations) with the deprecated Requirement Element are maintained (i.e. are NOT to be removed — removal from the diagram should not cause removal of the details of any pre-existing relationships / links between elements, which should continue to be maintained in the element’s metadata. Taking this approach supports reinstatement of the requirement if it is needed at a later date, as well as helping to understand its original context.
♦  Deprecated Requirements are not expected to be realised.
