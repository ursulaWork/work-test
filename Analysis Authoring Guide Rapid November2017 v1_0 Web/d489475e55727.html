Requirements Elements are used to articulate single requirements within an Analysis Model.
The following table describes the information that is held for each requirement element when produced as part of a formal requirement product.

Requirements Element: Name
A short description of the requirement.
After p in entry
The textual descriptions are included in the metadata as follows:
• 
The short name to be prefixed by a unique identifier; the format is <project shortcode>_ <unique number>_ <REQ>  followed by a short description.
• 
The identifier should be the same as the alias.
• 
The long textural description to include the priority in the text.
• 
Where an element is deprecated in which case the word ‘Deprecated’ MUST appear before the unique identifier.
After p in entry
Requirements: Notes
These notes fully describe the requirement.
After p in entry
The textual note should include:
• 
The priority in the wording uses  one of  ‘MUST’, ‘SHOULD’ or  ‘MAY’.
• 
Can  further classify any operational, legal, performance or reliability aspects etc.
• 
It can describe the decision for a requirement that has been deprecated.
• 
It can include references to any external documents, URLs  etc.

Requirement: Priority
The importance the stakeholders place on a requirement.  MoSCoW prioritisation is used.

The value should be the same as the requirement notes.

Requirements: Classification
Stakeholders involved in the requirement gathering:
• 
Originator —The person / organisation originating or raising the requirement, as well as the date it originates from.
• 
Sponsor — The person/organisation sponsoring or supporting the need for this requirement. 
• 
Contributors — The person / organisation contributing to the definition and resolution of this requirement and the date/s of their contribution.
