Purpose
After p in entry
The purpose of a Use Case is to describe a functional aspect of the domain / area under investigation:
• 
Describe a defined process, which in combination with other Use Cases and Actor roles describes the functionality to be built into the proposed system,
• 
Support the identification and gathering of Requirements for a domain model.

Description
After p in entry
A Use Case is a component within a Use Case Model, and is elaborated on a Use Case Diagram. It is used to represent a specific action that can be triggered in a system. It may be elaborated as a "white or black box use case dependant on the level of information needed by the audience for the domain.
After p in entry
Format & Content
After p in entry
Use Cases are represented as elliptical shaped elements on diagrams and are to be elaborated in the Use Case Catalogue Package.
They are identified during the Discovery stage of an analysis project.   
In the initial development a Use Case can have a brief name and identifier (as a candidate use case) and description.
A Use Case (and its element) can be further elaborated as:
• 
A ‘black box’ style Use Case: constraint information is added as the Use Case Model develops ensuring consistency across the model.
• 
A ‘white box’ style Use Case: include the constraints along with the processing steps for the Basic and Alternate paths:
◦ 
Basic Paths: to define the usual steps or events that the system needs to take to achieve the end Goal.
◦ 
Alternate Use Case Paths: define the processing to be undertaken when a known or planned event causes a deviation from the Basic Path.
• 
Composite Use Case: constraint information and hyper link to a supporting diagram  (Activity, Use case or Sequence diagram).
Constraint information for use cases includes: triggers, goal, pre and post conditions.
