The aim of an analysis project is to create a meaningful representation of the area under analysis to guide and inform subsequent activities such as design / build and testing.
The Data Products Rapid Methodology utilises an ‘Agile’ approach to project management in order to communicate these aspects to wider stakeholders, such as suppliers.
‘Agile’ is typically used in system development and helps teams respond to the unpredictably of building software through discrete incremental work items at a regular cadence (or rhythm), known as ‘Sprints’.
Analysis Models developed according to the Rapid Methodology are created and articulated using UML (Unified Modelling Language), and other accepted standards to provide a common understanding for all project participants.
- For details of the stages, roles, actions and products available as part of the Rapid Methodology, see the section: Rapid Methodology Overview.
An Analysis Model Template has been created that includes basic model / artefact outlines suitable to support this approach.
The development status for the artefacts within an analysis model may be defined as either: Optional / Required or Mandatory (the majority are Optional); there is considerable leeway as to which artefacts may be used by a specific project. The individual methodologies apply different constraints upon which artefacts are to be developed and how they are to be modelled; these are further defined by specific Adopted Notations and Product Descriptions, tuned to the approach being used.
Background to the Rapid Methodology
The Data Products Rapid Methodology was developed out of the need to adopt a more ‘Agile’ approach to project management and development (the Data Products also defines an older more Generic Methodology for the development of Analysis Projects).
