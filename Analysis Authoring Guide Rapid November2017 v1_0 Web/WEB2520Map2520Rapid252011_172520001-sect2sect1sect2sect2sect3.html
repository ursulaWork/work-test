The development team is dependant on content in the analysis model. A baselined analysis model is made available to the Core Team throughout the development lifecycle. Generally this is at end of each stage and crucially before a Sprint during the Delivery stage.
Who
Business Analyst 
Why (purpose)
This activity is to ensure the team is working on the correct content from the analysis model artefacts for a specific Sprint.
So that:
•  The Product Owner is aware of the content for a sprint.
•  The developers and testers are working on the same information.
•  The Backlog Repository is updated correctly.
•  There is a baseline record of the content for a Sprint.
When
Before a Sprint starts after the Sprint Planning completes.
Duration
3 hours.
What
The Business Analyst will check the contents of the Story Map for the Sprint and ensure that the acceptance criteria is correct and agreed with the Core Team.
The status of each element is updated in the model to "approved". The analyst should also ensure that the team are aware of any other analysis model artefact which may provide supporting information for the sprint.
The business analyst should:
1. Generate a report based on the content of the Story Map.
2. Create a baseline version of the analysis model.
This is made available to the rest of the core team for use in the Sprint.
The report is used by the Scrum Master to update the tickets for the Backlog.
Outputs
Analysis Model.
Story Map and supporting report.
Agile Framework
 Data Products specific analysis approach.
Hint: As part of the analysis model project set up, encourage developers and testers to use the Enterprise Architect tool so they can use a native version of the analysis model. Be prepared to "walk through" aspects of the analyst model with team members.
