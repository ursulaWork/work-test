Classifying and Organising (Requirements Elements and Artefacts) is key to managing content in the Requirements Catalogue package (further details are available in the section: Requirements for Rapid Methodology). 
Requirements can be organised by grouping them using sub-packages. This technique is useful for managing large number of Requirements Elements. The Requirement Diagram is used to visualise and organise the Requirements Elements in the package. 
Requirements can also be classified to indicate the type of requirement, the standard classifications used include:
•  Functional: — A ‘Functional Requirement’ describes what capabilities must be executed. This can be achieved using the sub-package technique and use of metadata. 
•  Non-functional: — A ‘Non-functional Requirement’ describes what constraints or quality characteristics apply. This can be achieved using the sub-package technique and use of metadata. 
Further classifications for requirements should be captured in the metadata for the Requirements Element, also the meaning of the classification should be documented to ensure a consistent application across the domain (CASE Toolset dependent) 
It is important to note that the contents of any sub-packages within the Requirements Catalogue Package cannot be mixed:
•  A Requirements Catalogue Package can only contain its classifying sub-packages and the Requirements Catalogue Diagram representing the requirements packages it contains.
•  A Classifying Package (Functional / Non-Functional) can only hold the individual Requirements Elements and the elaborating Requirements Diagrams for that type of requirement.
Complex projects:
Where a Project elaborates additional business or domain areas within the same Analysis Model, these should be held separately in uniquely named Requirements Catalogue Packages, within the top-level Requirements Package:
•  The Requirements Catalogue Package for each subset must include a Requirements Catalogue Diagram as well as organising sub-packages (e.g. Functional / Non-functional).
•  A business area / domain identifying name for the defined subset should be prefixed to the tile of the Requirements Catalogue Package, its Classifying sub-Packages and elaborating Requirements Diagrams.
Shared Requirements:
Where Requirements are shared across more than one defined domain / business area or subset within an Analysis Model, these should be held within a ‘Shared Requirements Catalogue Package’, created using the same organising approach (e.g. Functional / Non-functional Packages); however:  
The ‘Shared Requirements Catalogue Package’ does NOT require its own Requirements Catalogue Diagram; its Requirements Elements are elaborated on any of the other Requirements Diagrams within the Model that meet the same classification.
