Navigation Page Artefact (Adopted Notations)
Introduction oNext: Analysis Scope and Objectives Definition Document (Product Description)
This section outlines the Adopted Notations for the Navigation Page Artefact (for an example, see figure: example navigation page).
• 
The Navigation Page Artefact is a formal Artefact, integral to the analysis model (it does not have a life outside of it).
• 
The Navigation Page’s layout as well as its Model Structures and Organisational Packages are defined differently for each of the approved Methodologies.
NOTE: - Where Model Authors are NOT using the Analysis Model Template as the starting point for developing their analysis model — due to the complexity of its layout it is advised that the Navigation Page Package is export and re-import (as an XMI file) the from the template file into their model file (the correct Navigation Page layout will then be available for the chosen Analysis Methodology — Model Authors should also reset any lost link pathways, and set the imported Navigation Page as the Default Diagram for the Model (see the section: Navigation Page [EA Guidance] for further details).
• 
Outlines for each of the defined layouts for the Navigation Pages (according to each of the Analysis Methodologies) are predefined within the current Analysis Model Template
— these include:
◦ 
Navigation Page — Methodology Decision Page:
▪ 
Upon opening the Analysis Model Template for the first time the ‘Methodology Decision’ page enables the Model Author to choose the required Navigation Page for the Analysis Methodology that is to be adopted (detailed instructions for these actions are available within the template).
▫ 
The links on the chosen Navigation Page point to the specific Model Packages and Artefacts within the Template for the adopted Analysis Methodology approach. Once the Analysis Methodology and its Navigation Page have been chosen, it must always be set as the default model page. At this point any unwanted Navigation Diagrams / Packages (e.g. the Methodology Decision Page and Packages for any other Methodology) should be deleted.
• 
Navigation Page Model Information & Layout:
The following key aspects formally define the mandated layout and content for the Navigation Page within Analysis Models (for an example figure showing this layout, see the figure: example navigation page):
NOTE: - Where ‘links’ are referred to here, these mean any hyperlink-type process — the construction and use of which are CASE Toolset dependent.
◦ 
The Organisation Name and Logo: These should be displayed at the top of the Navigation Page (the organisation title is currently: NHS Digital), in accordance with the organisations’s approved publications format.
◦ 
The Model Name: This is placed at the very top of the page; it should show the Project and therefore the Model’s formal name, as agreed with the Model Owner (there is also a space for an additional sub-title if needed — otherwise this should be deleted).
◦ 
The Copyright Statement: detailing the organisation name, a copyright symbol and the year of issue — this should be displayed underneath the horizontal rule that is below the Model Title, and before any other Metadata information - this should currently read as:
‘Copyright © 2017 Health and Social Care Information Centre
’.
◦ 
Model Metadata: These are placed below the copyright statement and should include the following headings and content:
▪ 
Version No: The Model’s version control notation, in the format — ‘n.n’ (e.g. 1.4.2).
▪ 
Issue Date: The Models last formal Date of Issue.
▪ 
FileCM ID: For both the native CASE Toolset file and HTML generated output — as individual entries.
▪ 
Authors Name(s): The name of the Model’s Author/s (e.g. Model Custodian or Model Owner).
▪ 
Analysis Methodology: The name of the Methodology being utilised ( e.g. ‘Generic’ or ‘Rapid’).
▪ 
EA CASE Toolset used: The version of the EA CASE Toolset used to create the content — e.g. when the Model was last amended (where a different CASE Toolset is utilised the ‘EA’ should be changed to reflect the name of the Toolset).
◦ 
Key Project Stage / Phase Areas: These Areas (created as EA Boundary Regions) should be developed and titled for each of the key stages or phases of an Analysis Project (e.g. Planning / Scoping, Areas of Interest etc.,), as determined by the chosen Methodology:
NOTE: - The links and layouts of these areas are different for each of the Methodologies.
▪ 
These should include links to the Artefacts / Model Entities showing the products name (e.g. “Information Analysis Model”) within the bounded area (e.g. Areas of Interest):
▫ 
Where more than one instance of a Diagram type exists the name of the Artefact should be appended with a subset identifier (e.g. ‘Information Analysis Model — <<Scope Identifier>>’).
◦ 
Model Management & Documents Area: This is a distinct bounded area on the Navigation Page Artefact showing links to any internally / externally produced or related Documents or other non-diagrammatic content / Artefacts (as well as any non-EA diagrams) that are held within the Model:
▪ 
Where content is generated externally to the Analysis Model its content may also be embedde into the Model; this is CASE Toolset dependent (also see the section: Artifact / Document Element [EA Guidance]). This type of content is included for completeness and to ensure that any specific context or version referred-to is available as it was when the Model was published / distributed.
NOTE: - This usage can mean that embedded content may not be the most authoritative or up-to-date version.
▪ 
These types of Artefacts / Documents are typically held in the ‘Related Documents’ Package (within the top-level Methodology-named Model File); although these entities may also be held elsewhere in the Model hierarchy at the Model Author’s discretion (e.g. Discovery & Planning / Scoping Documents Packages):
▫ 
Model Outputs generated from outlines held within the ‘Master Documents Package’ of the Model may be re-imported into it, and placed in the ‘Related Documents’ Package; they should also have links to them placed within the ‘Model Management & Documents’ area of the Navigation Page.
▪ 
Each link should have a unique title (e.g. ‘Change Control’).
▪ 
The specific Documents that are expected to be included within a defined Analysis Model will depend on the Analysis Methodology that has been chosen, and as determined by its defined Artefact Set.
▪ 
Where a Document or Artefact cannot be held within the Analysis Model itself, a note giving details of where it may be sourced (e.g. a url / publication / organisation etc.,) can be added within the ‘External References Document’ (which is also linked-to within the ‘Model Management & Documents’ Area).
◦ 
‘Disclaimer’ Statement: The Approved Model Disclaimer Statement should be included within a bounded area at the foot of the Navigation Page — this is currently defined as:
▪ 
“IMPORTANT NOTE: This model has been developed in consultation with the design teams of NHS Digital and its contracted suppliers. The behaviour and utilisation of services has been modelled in accordance with information given by these teams and the models do not in themselves constitute design guidance or requirements”.
• 
Acronyms should not be used within the Navigation Page or in any of its ‘link’ references.
• 
Diagram notes on the Navigation Page should only be added in accordance with the chosen Methodology.
• 
If the name, organisation or contents of Packages are changed during the development of the Analysis Model then the Navigation Page and its links must also be modified as necessary to reflect these changes.
