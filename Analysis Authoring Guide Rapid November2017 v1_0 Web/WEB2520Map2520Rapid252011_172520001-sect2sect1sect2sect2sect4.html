Models are authored according to a defined set of roles. The author of an Analysis Model is usually an analyst. The lead analyst for a project will also usually take on the role of Model Custodian. Authors should ensure that where they work on an analysis model that their full name (not shortcodes) is recorded against the work that is completed by them. Models maybe authored by a single analyst or by several, using a distributed authoring approach. Employing multiple model authors will require a process or functionality to coordinate the various copies of the model files, as well as to maintain rigorous version control. One approach that can be employed to manage the different ‘copies’ of a file, as well as to handle their eventual reintegration prior to publication is to use a content management software, such as Subversion (SVN).
- There maybe some variation in the responsibilities allocated to each role where a project is subject to multiple author inputs, or where there are authoring inputs across different teams.

Collaborative Working
Collaborative Working
Where a model is authored by a number of different authors, either within Data Products or externally to it (e.g. where using analysts who are members of other teams) is to take a distributed authoring approach. To control a distributed approach the activities undertaken by the Model Custodian and the various authors needs to be clearly defined.
The custodian should divide the model and allocate parts to the various authors. The partition of a model can be achieved using defined structural model packages, such that each package creates a partition that can be more easily exported from / imported into the definitive Model held by the Custodian.
Subversion (SVN) can also be used to manage different versions of a Model being worked on by multiple Authors. The Model is first ‘branched’ and a specified branch is allocated to each Model Author. At the end of the project (or before a specified project stage ends), the model custodian would decide which aspects to incorporate back into the ‘definitive’ Model prior to publication.
Authors should only work on the content allocated to them, before returning it to the custodian. It is the custodians role to reintegrate and consolidate returned packages into the definitive model (ensuring model consistency / integrity); this will lead to an updated definitive version of the model, which must be entered into the Model Library.
Taking a Distributed Authoring approach can cause complication if more than one person needs to work on the same part of the model or artefact. It is strongly advised that each author work ONLY on one part of the model at any one time; any comments / suggestions / requirements for amendments to other parts must be communicated to the custodian for separate work / consolidation
It is the role of the model custodian to allocate which part/s of a Model is/are to be worked on. Each ‘part’ can then be exported, worked upon and re-integrated into the definitive model as a self-contained package. A complication can arise where content has links outside of the ‘part’ being worked on. Where work to amend this type of link is required, analysts should communicate the requirement back to the model custodian to authorise before they are undertaken, so that any elements / links can be checked and where necessary reinstated if they are lost upon consolidation.
Dividing a model into different ‘branches’ to be worked on by different authors can reduce reintegration issues (as well as being used to create different instances of a model e.g. for ‘as is’ or ‘proposed’ versions).
Model parts distributed for work by multiple authors are still subject to Local Version Controls, and overseen by the model custodian.
- A persons full name should be used to identify model authors within models (not shortcodes).
