Understanding Actors
Understanding Actors
‘Actors’ exchange data with a system. An Actor can be a human user, external hardware, or another system.
To fully understand the system's purpose you must know who the system is for, that is, who will be using the system. Different user types are represented as Actors.
The difference between an Actor and an individual system user is that an Actor represents a particular class of user rather than an actual user. Several users can play the same role, which means they can be one and the same Actor. In that case, each user constitutes an instance of the Actor. However, in some situations, only one person plays the role modelled by an Actor. For example, there may be only one individual playing the role of system administrator for a rather small system. The same user can also act as several Actors (that is, the same person can take on different roles).
Actors may be represented differently depending upon the diagrammatic context for their use; on a Use Case diagram they are represented as a stick-man element, whereas on an Activity Diagram they are represented by the use of Swimlanes. Details of specific annotations are provided as part of the definition and core notations for each diagram type.

Identifying Actors
Identifying Actors
Consideration must be given to the individuals who will use the system (system users) and to how they might be categorised. Using scenarios will help in this respect. The following set of questions are useful to have in mind when deciding who or what are your Actors:
•  Who will supply, use, or remove information?
•  Who will use this functionality?
•  Who is interested in a certain requirement?
•  Where in the organisation is the system used?
•  Who will support and maintain the system?
•  What are the system’s external resources?
•  What other systems will need to interact with this one?
There are several different aspects of a system's surroundings that you will represent as separate actors:
- Identifying the Actors also means that you establish the boundaries of the system, which helps in understanding the purpose and extent of the system. Only those who directly communicate with the system need to be considered as Actors. By including more roles than that in the system's surroundings, you are attempting to model the business in which the system will be used and not the system itself.
•  Users who execute the system's main functions.
•  Users who execute the system's secondary functions, such as system administration
•  External hardware the system uses.
•  Other systems interacting with the system.
