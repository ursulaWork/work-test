This section outlines the Project Roles and Titles for Analysis Projects undertaken by Data Products within NHS Digital.
The development of Analysis Models and their artefacts / products in accordance with the Rapid Methodology requires the involvement of a range of team members / roles; these are fully described in this section.

Agile Project Team Rolesspacer text
An ‘Agile’ project is made of a number of team members. Some members will be involved throughout the project life cycle, whilst others may only participate during initial project setup or for specific approvals. Collectively these roles are referred to as the ‘Agile Development Team’.
The main roles and key responsibilities include:
NOTE: - This is an exhaustive list of roles. Often on smaller projects some of the roles will devolve to whoever is assigned and / or available (and often falls to the Core Team).
Project Manager:
• 
Manages the project on a day to day basis on behalf of the Project Executive.
• 
Chairs/facilitates initial project setup & scoping meetings and release planning meetings.
• 
Escalates risks/issues from the project team to the Project Executive.
Project Executive:
• 
Accountable for Project delivery.
• 
Budget Holder.
• 
Decision maker on all changes within budget.
• 
Decision maker on solution choices.
• 
Provides authority to proceed on new stages of delivery.
Product Owner:
• 
Represents the customer (i.e. all stakeholders) within the team, in particular clarifying details, setting priorities amongst work items and reconciling scope with business constraints. The Product Owner must be empowered by the Business Owner to make such decisions on behalf of customers.
Specific responsibilities of the Product Owner include:
• 
Owns user requirements.
• 
Accept/rejects work items within sprints.
• 
Initial point of consultation and authority for Analysis deliverables as well as consultation for Analysis deliverable content.
• 
Prioritises requirements and work items:
1.
At the beginning of the project to decide what's in and what's out of scope of the project (roadmap).
2.
During each of the sprints to decide what goes into each sprint (sprint planning).
3.
At the end of each sprint to give feedback.
Analysis Manager:
• 
The manager only reviews the forecast. The analyst themselves will be in charge of forecasting their time:
• 
Support and escalation route for Analyst.
• 
Allocates Analysis resources.
Analyst (Business Analyst):
The Analyst role (Business Analyst) is required throughout the product lifecycle/project, and is an integrated part of the ‘Agile Development Team’:
Specific responsibilities for the Analyst Include:
• 
Undertake activities to shape and scope the project.
• 
Produces Analysis specific deliverables for the project:
◦ 
Analysis scope and objectives document.
◦ 
Analysis model artefacts.
• 
Refine and elaborate analysis model artefacts so they can be understood and prioritised.
• 
During sprints supports the core team.
The Analyst reports to the:
• 
Project Manager,
• 
 Analysis Manager (by exception):
◦ 
Exception rule is usually enforced on smaller projects when a full Project Manager is not required/available
Development Team Manager:
• 
Acts a functional team manager assigns resources to the development team.
• 
Monitors measures and reports.
Developer/s:
• 
Develop software based on Analysis deliverables, in line with customer need.
• 
Help size User Stories as part of Sprint planning: Story -> Task -> Man hours.
• 
Adapts process.
Test Manager:
• 
Manages test process.
• 
Plans for testing activity.
• 
Manages User Acceptance testing.
Tester/s:
• 
Produces and executes test scripts based on Analysis deliverables.
UI Designer/s:
• 
Develops UI based on requirements.
Scrum Master:
• 
Govern the scrum process and ensures it is being followed.
• 
Chairs / facilitates Sprints, retrospectives and sprint planning meetings.
• 
Remove any impediments that obstruct a project team’s pursuit of its sprint goals.
• 
Champion agile values and behaviours.
User/s:
• 
Consulted on Analysis activity to help the Analyst develop artefacts.
• 
Users are represented on the project via the Product Owner role.
Infrastructure Specialist:
• 
Consulted on Analysis activity - specifically; awareness of current hardware and support requirements.
• 
Supports Implementation Planning.
• 
Conducts Implementation of approved software builds.
Analysis Technical Assurance:
• 
Reviews Analysis deliverables against the Analysis Methodology — making recommendations to the Analysis Manager as well as to the Analyst.
• 
External to project team.
Architect:
• 
Is consulted during Planning & Discovery stage to draft a solution compliant with the technical architecture and best practice. With later involvement on ‘an as and when required’ basis.
UX Designer:
• 
Assists in producing prototypes (wireframes through to working models) to drive user / product owner elicitation of requirements and incremental improvements.
User Researcher:
• 
Uses User Feedback, Information Analysis and Operational Research techniques to clarify / drive out user requirements.
Concept of the Core Teamspacer text
Some aspects of an Agile project only require input from Core Team members, which is a subset of the Agile Development Team listed in the section: Agile Project Team Members; it includes.
• 
Product Owner,
• 
Business Analyst,
• 
Developer,
• 
Tester,
• 
Scrum Master.
