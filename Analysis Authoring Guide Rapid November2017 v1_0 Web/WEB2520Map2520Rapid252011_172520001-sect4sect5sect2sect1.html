This section outlines the Adopted Notations for the Requirements Catalogue Package Artefact
•  The Requirements Catalogue is a package within the top-level ‘Requirements’ Package of the analysis model:
–  If it is a Requirements Artefact and is considered a formal artefact for review and approval purposes.
–  The Requirements Catalogue Package contains:
♦  A Requirements Catalogue Diagram (as a package diagram) — provides an overview of the packages within the catalogue (and optionally their element / sub-package contents — CASE toolset dependent)
♦  A series of classifying sub-packages.
- Further localised classification of a requirement can be applied to the metadata within its Requirements Element (e.g. ‘Expressed’ / ‘Derived’ etc.,).
◊  A Requirements Diagram - elaborating the requirements for that classification
◊  All of the individual Requirements Elements for that classification (except for complex models where a sub-package holding the ‘shared’ Requirements Elements should also be used).
, containing:
◊  A Requirements Diagram — elaborating the requirements for that classification
◊  All of the individual Requirements Elements for that classification (except for complex Models where a sub-package holding the ‘shared’ Requirements Elements may also be used )
•  Its contents can be represented diagrammatically using:
–  A Package Diagram (recommended for complex models).
–  A Requirements Diagram Artefact.
•  Complex model packages:
–  Where more than one business subset is elaborated within the same Analysis Model separately named Requirements Catalogue Packages should be created within the top-level Requirements Package. Each of the subset packages should have its own Requirements Catalogue Diagram (except where they are part of a ‘shared’ requirements sub-package), and the business subset name must prefix the titles for its sub-packages and diagrams.
