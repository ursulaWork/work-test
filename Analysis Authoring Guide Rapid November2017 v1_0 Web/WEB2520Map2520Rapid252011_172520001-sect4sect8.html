Heading

Product Name
Activity Diagram Artefact — (AD)
Status
The status of the Activity Diagram Artefact is ‘Optional’.
Purpose
The purpose of the activity diagram is to describe the dynamic behaviour of a process (business or system). It can support the identification of:
•  Use Cases and the relationships between use cases.
•  User stories and the relationships between user stories and how they "fit" within a business process.
It can also be used to describe a process within a user story or use case to allow it to be further detailed elaboration/analysis.
Audience & Use
Internal:
•  Business Analysts: as an analysis technique to aid verification of functionality and/or process activities.
•  Project Team: to meet stated Purpose for this Artefact.
•  Test Team: to inform test cases.
•  Technical Architects: to inform technical design.
External:
•  All: Starting point for process view of the domain (unless there is a pre-existing Stakeholder Business Model).
•  Domain Experts: verify the business processes and requirements.
Description
The Activity Diagram Artefact is a UML Activity Diagram that describes the dynamic behaviour of a process (business or system), to aid verification of functionality and / or process activities.
Format & Content
The Activity Diagram Artefact comprises a UML Activity Diagram and associated Activities within the package structure. The Activity Diagram Artefact represents Activity elements and the Flows between them.
It is developed during the Discovery Stage of a Project.
It is held within the model structure as follows: in a package named Activity Diagram Artefact in the ‘Rapid Analysis Methodology’ package:
•  Where the model contains more than one Activity Diagram Artefact, these should be held in a suitably named sub-packages within the Activity Diagram Artefact package.
This artefact is created using the currently approved CASE toolset (Enterprise Architect) - content not included at this time.

Technical review- using review checklist
Activity Diagram Artefacts are held in the package named: Activity Diagram Artefact.
Technical review- using review checklist
A sub-package structure is use where more than one Activity Diagram Artefact is created.
Technical review- using review checklist
All ‘normal’ activities and actions (i.e. exclude initial & final activities) must have one incoming flow and one outgoing flow.
Technical review- using review checklist
Actors are represented as Swimlanes in the Activity Diagram and are also represented by Actors within the Actors Catalogue Package.
Technical review- using review checklist
Individual Activity Elements should describe a single discrete, complete, and testable process.
Technical review- using review checklist
Individual Action Elements should describe a single discrete, complete, and testable step in a process (action). There are no composite actions associated with an action allowed.
Technical review- using review checklist
Object flows must be associated with ‘interrupt events’ catering for ‘duplication’ and ‘timeout’ events. The events and the object flow must be encapsulated within an interruptible region.
Technical review- using review checklist
From a Decision point there must be at least one alternate path.
Alternate paths will be denoted as [Alt_n – <<guard condition>>], where ‘n’ is an incremental integer for every decision point (e.g. Alt 1, Alt 2, Alt 3).
The incoming flow must be repeated as one and only one of the out-going flows (e.g. if Alt_1 in, then one of the out-going flows must be Alt_1).
