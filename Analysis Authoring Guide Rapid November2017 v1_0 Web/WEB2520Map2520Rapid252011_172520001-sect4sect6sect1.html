Heading Details
Product Name/s
User Story
Description
User Stories are short, simple descriptions of a feature told from the perspective of the person who desires the new capability, usually a user or customer of the system.
Status
The development status of User Story elements is ‘Required’.
Purpose
The purpose of User Stories is to agree with the Product Owner the functionality needed so that the User Story can be added to the product backlog for development.
Audience & Use
Internal
•  Product Owner: Identify User Stories.
•  Analyst: Document and elaborate User Stories adding more detail and relevant acceptance criteria.
•  Developers: Provides guidance on estimating development time.
•  Test Team: to inform test cases.
External:
•  Domain Experts: Verify against other artefacts.
•  Suppliers: To use as input into development work.
Format & Content
User Stories are individual elements in the analysis model written in the syntax:
As a [role]:
I MUST / SHOULD / MAYto: [action]
So that: [goal]
They are created during the Discovery Stage and developed iteratively during delivery, and need to be completed as part of planning a Sprint.
They are held within the following model structure package/s: In the User Story Catalogue package.
They are created using the currently approved CASE Toolset.
Construction Guidelines
•  For details of how to construct a User Story, see the section: User Story Adopted Notations.

Quality Criteria
Quality Requirement
Quality check
 
The User Story is independent and of a size suitable to build in a single sprint.
Verified by development team.
 
The User Story will add value to the product.
Verified by product owner and development team.
 
Terminology of User Story content is recognisable to stakeholders.
Verified by product owner.
 
User Story elements conform to the approved Adopted Notations.
Technical checklist.
 
The User Story is testable
Verified by test team.
 
User Story elements are stored in the User Story Catalogue.
