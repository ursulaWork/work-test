Where working with Agile type projects, it is accepted that changes will be expected in a product as it develops. This presents a challenge to the analyst in that they need to manage elements in the analysis model:
1.
Elements are identified, classified and organised within the model according to their type and the methodology,
2.
Changes to the model are managed throughout the project and communicated to the project team as part of the Rapid process,
3.
Traceability techniques are used to ensure consistency across the model as it develops.

Deprecating Elementsspacer text
Redundant elements should be ‘deprecated’ and NOT deleted as they may be useful for historical reference or traceability.
This is particularly true of elements such as: Requirements, Use Case and User Story Elements that have been agreed with the Product Owner. It is also recommended to keep the project team informed when elements are deprecated.
An item is ‘deprecated’ to remove it from an active status within the model whilst retaining its presence for backwards compatibility, future development, or to aid understanding of its historical context. Inactive items / elements retained for future reference should always be labelled as ‘Deprecated’.
Requirements Elements for ‘deprecation’ are to be moved to a top-level ‘Deprecated’ package, and all of the rules and guidelines for deprecating model entities, as well as the specific Adopted Notations for deprecating Requirements and User Story elements SHOULD be adhered to.
