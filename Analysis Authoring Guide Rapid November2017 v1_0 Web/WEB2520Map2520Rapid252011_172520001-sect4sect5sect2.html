•  Suppliers: use as input into development work
•  Users: use as input to User Acceptance Testing
Description
The requirements catalogue is a package within the analysis model.It containsclassifying sub-Packages to hold the individual Requirements Elements, and the Retirements Diagram
Format & Content
The Requirements Catalogue is a Requirements Artefact modelled according to its own Adopted Notations. It is developed using the currently approved CASE Toolset (see: Enterprise Architect [EA].
It is held within the top-level ‘Requirements’ package.
A Requirements Catalogue may also be produced as a RTF report for external use but this is not the formal deliverable.
Its development commences during the Inception Sage of a project, and continuesover the lifecycle of the project.
Construction Guidelines
The construction of the Requirements Catalogue Package Artefact is defined by its Adopted Notations.
Template(s)
Outline package structure in the Analysis Model Template with a predefined template generator available within the Analysis Model Template RTF report generator.

Quality Criteria
Quality Requirement
Quality check
The terminology of content must be recognisable to domain Subject Matter Experts.
Domain expert review.
The requirement catalogue content agrees with the scope of the project.
Project manager review
Conforms to the approved modelling approach as well as the adopted notation.
Technical review – using reviewchecklist
Each package must contain an adequate description of its purpose in its metadata ‘Notes’ section.
Technical review – using reviewchecklist
Requirements Catalogue package is to be created as a root-folder / package at the top level of thedomain analysis model, as a single package / folder.
Technical review – using reviewchecklist
The Requirements Catalogue, and content at any level within a it, must set their titles to a uniquedescriptive name.
Technical review – using reviewchecklist
All the packages / folders within the Requirements Catalogue Package must include a RequirementsDiagram.
Technical review – using reviewchecklist
For complex domains where more than one business /domain subset of requirements are developed, each of them must be held in a discrete and uniquely labelled Requirements Catalogue Package.
These must each contain their organisingsub-packages (e.g. Functional/ Non-functional) within them — each of these sub-packages are to be elaborated with separate Requirements Diagrams.
Technical review – using reviewchecklist
Further quality criteria for Requirements Elements is available in the Requirements Diagram Product Description.
