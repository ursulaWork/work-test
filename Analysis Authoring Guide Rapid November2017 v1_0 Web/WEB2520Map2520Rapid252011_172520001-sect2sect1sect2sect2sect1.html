This section outlines the Version Controls and related approaches for Analysis Models and their artefacts.
The methodologies define their approaches to version control for Models and their Artefacts as either ‘formal’ or ‘local’. These controls maybe applied either to the Model as a whole or individually to each of its Artefacts / Documents. In addition a change history log is used to record any changes that may result in a new / up-issued version of a Model. Version Control also governs some aspects of the naming of Model Files.
- Where any part of an Analysis Model is circulated outside of Data Products, Authors should ensure that the correct version control information is recorded on the circulated models.

Formal Version Control
Formal Version Control
Formal Version Controls for Analysis Models are managed as part of NHS Digital’s File and Document Management System (this currently utilises the tool: FileCM); this is the organisation’s repository for all content released for publication or external distribution (for other forms of version control see the section: Local Version Controls).
- All Analysis Models held within FileCM must also adhere to all of the formal model publication and output rules for Analysis Projects.
All Analysis Model Artefacts (as well as any other formal Documents / Artefacts etc.) must adhere to the rules and conditions surrounding the use of FileCM, including an ID. The ID classifies the content within FileCM as part of the Data Products Model Library, as well as including a version identifier.
Version identifiers for Files are created upon the initial upload of a Model File to FileCM; these are automatically amended by by the tool whenever a File is up-issued, adding a further increment to the final two digits. (e.g. up-issuing file: ‘NPFIT-SHR-MODL-GUID-0014.04’ changes its ID to ‘NPFIT-SHR-MODL-GUID-0014.05’).
- Where Distributed / Multiple Authoring is employed to develop Analysis Models, Local Version Controls may also be temporarily used in addition to the File’s ID (prior to publication and outside of FileCM), under the control of the Model Custodian.
Local Version Control
Local Version Control
Where distributed / multiple authoring is utilised to develop Analysis Models, the Model Custodian may use a Local Version Control approach to identify versions of the Model file authored by different people.
The following outlines the rules for their use:
•  Local Version Control should only be used by Authors during a Model / Artefact’s development (e.g. to support reviews or for distributed / multiple authoring).
–  Local Version Control identifiers should take the form of an addition appended to the FileCM ID and should only be recorded within a Model File as part of its own metadata — NOT as part of its formal File ID
–  Local Version identifiers are superseded in ALL cases by the formal File ID (as part of the Formal Version Control Approach) — and MUST be removed from Model Files prior to their being published / uploaded / up–issued to the Model Library / FileCM
–  Formal File ID version numbers within the Model Library / FileCM may not always coincide with Local Version Control identifiers; any inconsistencies between Local and Formal Version Identifiers must always be resolved and Local Identifiers removed by the Model Custodian prior to uploading / up-issuing the Model to the Model Library / FileCM
•  Local Version Control for Models or Artefacts should use a numbering system that is an addition to and after the formal file ID:
–  On the navigation page of a model a separate ‘Version No’ heading should be created at the top of the page, against which any Local Version numbering should be recorded:
♦  n1: the Main version number indicates significant changes in the models status or contents
◊  For initial drafts of a model, n1 is set as 0 (i.e. 0.1 to 0.n), n1 is incremented to 1 (e.g. 1.0) on model approval
♦  n2: the Minor version number indicates small changes in the models status or contents
♦  To identify Multiple authors it may also be useful to also append their shortcode as a form of Local Version Control, after any version identifiers - e.g. ‘FormalFileID_0.5_jobl’ during development
♦  When a Model is ‘branched’ within a Subversion (SVN) Repository to support Distributed Authoring:
◊  Where different versions of a Model are developed using SVN (e.g. for ‘as is’ and ‘proposed’ versions of a Model), each should have details of the version type added to its file identifier: e.g. ‘FileName_n2.1_Proposed’
◊  A ‘branched’ version of a model would not normally be formally published (e.g. it would not have its own FileCM ID); unless the versions require formal publication (e.g. for traceability or parallel development within the project’s scope) — where it is necessary to formally publish a branched Model, each version should remain related to the ‘base’ Model during development, and only becomes a distinct Model in its own right with an individual FileCM ID, at the end of the project (otherwise any ‘shared’ components or mutual links could be lost)
Model Changes - History and Log
Model Changes - History and Log
One of the main reasons for up-issuing or creating new ‘versions’ of Models are when changes are made to the content. Part of managing a Models’ Version Control is ensuring that a record is kept of all changes to it; this formally utilises the Model Change History Log. The Model Change History Log may also developed and maintained outside of the Model.
The following aspects of this approach should be understood:
•  The formal approach defined by the Methodology uses a Model Change History Log (available as a template from FileCM using ID: NPFIT-SHR-MODL-GUID-0005, to create a document which details specific changes made to the Model. These are captured for every Model version increment (the Model is then ‘up-issued’ according to either the formal or local Version Control mechanisms). The Log should always be embedded into the Model:
–  It should be held in the Related Documents Package and must be made a selectable item within the Model Management and Documents area of the Navigation Page.
•  As an informal process where a CASE Toolset is being used:
–  Authors should note that the approach defined by the chosen methodology takes precedence, and any notable changes must always be transferred and recorded in the Model Change History Log.
–  Any changes recorded using a toolset’s change recording processes must be transferred to the formal Change History Log, and then removed from the Model prior to publication.
•  Formally Publishable Artefacts may be subject to Project Controls; and are only to be published / distributed as an integral part of the Model File

Model and File Naming
Model and File Naming
When creating a Model / Project File for the first time, it should be named using the name (or accepted short-name) for the Analysis domain (e.g. a file titled for ‘Mental Health’ might also be referred to as ‘MH’).
Where the Model revisits Analysis for an old Project then either the existing file title maybe used (or amended to reflect current nomenclatures — e.g. where the project name has changed). If the same Project name is used for a new file then a version number for the file should be added as a suffix to the File ID (for details see the section: File ID meaning).
- Where new analysis is being continued using a pre-existing Analysis Project file, then any new work should be clearly identified within the Model using identifiers added to folders / diagrams, as appropriate.
